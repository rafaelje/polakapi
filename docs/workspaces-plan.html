polakapi · plan de diseño

Workspaces & Projects · contexto por proyecto

Un plan para introducir workspaces (espacios de trabajo) que agrupan projects (carpetas reales en disco). Cada proyecto tiene su propio set de terminales, sus propias notas y, eventualmente, sus propios agentes. Cambiar de proyecto cambia la vista — pero todos los procesos siguen corriendo en paralelo: nada se pausa, nada se mata. El proyecto que dejaste sigue compilando, sirviendo, testeando, mientras vos mirás otro.

01Visión general

Hoy polakapi tiene un set global de terminales y notas. La meta es que el usuario pueda saltar entre proyectos y la app recuerde — y restaure — todo lo que estaba haciendo ahí.

Workspace

Carpeta lógica de proyectos relacionados. Ej. Boom, Sockeye. Sólo nombre — no tiene path.

Project

Una ruta real en disco más metadata: nombre, ícono, color. Es la unidad activa.

Contexto

Terminales + notas + (futuro) agentes. Vive dentro de un proyecto.

Jerarquía conceptual

02Principio: ejecución en paralelo

Regla de oro Los proyectos nunca se pausan al perder el foco. Cambiar de proyecto o de workspace es puramente un cambio de vista. Todos los procesos — pnpm dev, cargo watch, docker compose, un agente ejecutando — siguen corriendo en background, leyendo y escribiendo a sus PTYs, hasta que vos los matés explícitamente.

Esto es lo que diferencia a polakapi de un editor que “abre” un proyecto a la vez. El modelo mental correcto es:

✓ Sí es

  • Un panel de control de N proyectos corriendo a la vez.
  • El proyecto activo es el que ves.
  • Cambiar de vista = swap de DOM, no de procesos.
  • Cerrar la app = matar todo (con confirmación si hay vivos).

✗ No es

  • Un editor donde “abrís” un proyecto y “cerrás” otro.
  • Los procesos no se suspenden al cambiar de proyecto.
  • Las terminales no se respawnean cuando volvés.
  • No hay “proyecto cerrado pero con procesos” — si tiene procesos, está vivo.

Diagrama: lo que pasa cuando cambiás de proyecto

Implicaciones concretas

Acción ¿Qué pasa con los procesos?
Click en otro proyecto SIGUEN VIVOS Sólo se desmonta el DOM xterm del anterior. El PTY sigue recibiendo output, que se acumula en el buffer del terminal.
Cambio a otro workspace SIGUEN VIVOS Idéntico al caso anterior, no hay diferencia entre cambiar dentro o entre workspaces.
Volver a un proyecto que dejaste hace 20 min VES EL HISTORIAL El buffer de xterm tiene todo lo que el proceso escribió mientras no mirabas (scrollback completo).
Cerrar el sidebar / minimizar la app SIGUEN VIVOS Sin cambios. Es solo UI.
Eliminar un proyecto con procesos vivos CONFIRMACIÓN Modal: “Tiene N procesos corriendo.” Si confirmás, se matan todos los PTYs de ese proyecto.
Cerrar la ventana / quit de la app CONFIRMACIÓN GLOBAL Si hay procesos vivos en cualquier proyecto, modal con la lista antes de matar. Sin confirmación si no hay nada vivo.
El indicador de vida importa más que nunca Como hay procesos invisibles corriendo, el badge de “N terminales” en el sidebar es crítico. Debería incluir además un punto de color que distinga: ● activo con output reciente · ● idle · ● proceso terminó con error mientras no mirabas.

Costo del modelo

  • Memoria: cada xterm instance + buffer pesa ~5–15 MB; cada PTY proceso es lo que pese su proceso (un pnpm dev puede ser 200–500 MB). 4–6 proyectos activos = realista. 50 = no es el caso de uso.
  • CPU: los terminales ocultos no renderizan nada — su xterm está desmontado del DOM. Solo el PTY consume CPU si su proceso lo consume.
  • Recursos del sistema: puertos ocupados, archivos abiertos, locks — son responsabilidad del usuario. polakapi no intenta ser “smart” cerrando cosas; si tenés dos dev servers en el puerto 5173, uno va a fallar y vas a verlo en su terminal.

03Modelo de datos

Estructura mínima. Se persiste como JSON en el store de Tauri (mismo plugin que ya usás para layout). 

// src/modules/workspaces/types.ts
export interface Workspace {
  id: string;          // uuid
  name: string;        // "Boom"
  color?: string;      // tinte opcional para el chip
  collapsed?: boolean; // estado UI
  projects: Project[];
}

export interface Project {
  id: string;
  name: string;        // "Multiverse Admin"
  path: string;        // "/Users/rafaelje/projects/boom/clients/sport-admin"
  icon?: string;       // emoji o letra inicial
  notes?: string;      // contenido del panel notes
  terminals: TerminalSpec[]; // config, no estado vivo
}

export interface TerminalSpec {
  id: string;
  title?: string;     // "dev server", "tests"…
  shell?: string;     // override del shell por defecto
  cwd?: string;       // default: project.path
  startupCmd?: string;// comando opcional auto-ejecutado al spawn
}

export interface WorkspacesState {
  workspaces: Workspace[];
  activeProjectId: string | null;
}
Por qué TerminalSpec y no “sesión viva” Las sesiones PTY no sobreviven a un reinicio: cuando matás el proceso, se va. Lo que persistimos es la configuración de qué terminales abrir cuando el proyecto se active (cwd, comando opcional, título). El estado vivo (PID, buffer, scrollback) vive en memoria mientras la app corre.

05Flujos: crear, editar, eliminar

Añadir workspace

Fricción mínima Click en [+] del header del sidebar → aparece un input inline al inicio de la lista → escribís el nombre → Enter guarda, Esc cancela. Sin modal.

Añadir proyecto

Dos caminos, mismo resultado:

  • Hover en un workspace → aparece ＋ Añadir proyecto… debajo.
  • Drag & drop una carpeta del Finder sobre el workspace.

Al clickear se abre el file picker nativo (tauri-plugin-dialog) en modo carpeta. Una vez seleccionada:

Editar

Doble-click sobre el nombre = rename inline. El menú ⋮ ofrece acciones extras:

Acción Workspace Project
Renombrar ✓ (doble-click o ⋮) ✓ (doble-click o ⋮)
Cambiar color / ícono ✓ ✓
Cambiar ruta — ✓ (re-abre picker)
Duplicar — ✓ (mismo path, nuevo nombre — útil para “sandbox”)
Mover a otro workspace — ✓ (drag & drop o submenú)
Eliminar ✓ (confirmación si tiene projects) ✓ (confirmación si tiene terminales vivas)

Eliminar — niveles de fricción

Proyecto sin terminales vivas Click eliminar → toast con Deshacer durante 5s. Sin confirmación.
Proyecto con terminales vivas Modal con resumen: “Tiene 3 terminales corriendo. Se cerrarán todas.” Botón Eliminar destacado pero no por defecto — el botón por defecto es Cancelar.

Mismo patrón para workspace: si tiene projects, modal explícito; si está vacío, undo-toast.

06Cambio de contexto

Cuando hacés click en un proyecto, todo el espacio central, las notas y (futuro) los agentes, cambian. Los procesos del proyecto anterior siguen corriendo — ver § 02 Principio: ejecución en paralelo. Solo se desmonta el DOM xterm del anterior; los PTYs siguen recibiendo output, el cual se acumula en el buffer del terminal para cuando vuelvas.

Detalles importantes

  • El cambio es síncrono y barato. No matamos PTYs, no recreamos xterm instances — solo intercambiamos qué contenedor está visible.
  • Las notas se guardan al salir. El textarea dispara save cuando el proyecto deja de ser activo.
  • Si el proyecto destino no tiene terminales abiertas todavía, se auto-spawn las definidas en terminals[] (si las hay) o se queda vacío con el botón + Terminal visible.
  • “Live indicator” en el sidebar: badge con número de terminales corriendo del proyecto, aunque no esté activo. Da paz mental.

07Terminales por proyecto

Hoy TerminalManager es global. Hay que refactorizarlo a uno-por-proyecto. La pieza clave: nunca destruimos el manager — sólo cambiamos cuál está montado.

Estrategia de montaje TerminalManager recibe su DOM root en mount(root) y se puede unmount() (preserva las xterm instances en memoria). El router llama unmount(oldProject); mount(newProject, gridRoot) al cambiar. xterm.js soporta bien esto con terminal.open(newContainer).

“Run command in all…” — alcance

El botón actual ejecuta el comando en todas las terminales globales. Con workspaces, debe ejecutar sólo en las del proyecto activo. Si querés cross-project, hay que pensarlo aparte (probablemente no es lo que se quiere por defecto).

08Persistencia

Usar el mismo plugin @tauri-apps/plugin-store. Dos archivos separados para evitar acoplar dominios:

Archivo Contenido Frecuencia de escritura
layout.json (ya existe) Anchos de paneles, toggles, cols del grid Al cambiar UI (debounced 250ms)
workspaces.json nuevo Workspaces, projects, terminals specs, activeProjectId Al CRUD + cambio de proyecto (debounced)
notes/<projectId>.json nuevo Contenido del textarea por proyecto Debounced 500ms mientras escribís

Las notas en archivo separado para no reescribir el JSON gordo de workspaces en cada teclazo. Alternativa: dejarlas inline en Project con debounce más agresivo.

Migración del estado actual El notesContent que hoy guarda layout.json debería migrar a un proyecto “Default” o quedarse como notas globales del workspace por defecto. Vale la pena decidirlo antes de la fase 1.

09Plan por fases

Cada fase termina en un estado funcional y commiteable. No hace falta hacer todo de una vez.

Fase 1 Modelo + CRUD + reestructura del toolbar

Objetivo: poder crear/listar/editar/eliminar workspaces y projects. Y dejar el toolbar global limpio.

  • Crear src/modules/workspaces/ con types.ts, store.ts, workspaces-panel.ts.
  • Implementar render del árbol en el sidebar izquierdo (reemplaza el placeholder).
  • Orden alfabético por defecto al insertar. Drag & drop entre hermanos y entre workspaces, que setea order explícito.
  • Añadir comando Rust fs_validate_path + tauri-plugin-dialog para el file picker.
  • Validación al crear/editar: path debe existir, ser directorio y ser legible. Sin path = no se puede guardar.
  • Persistencia en workspaces.json + carga al startup.
  • Empty state cuando no hay proyecto activo + restauración del último activo al arrancar (si su path sigue siendo válido).
  • Reestructurar toolbar: sacar + Terminal, Run in all…, Cols del toolbar global. Por ahora caen en el mismo lugar pero condicionados a “hay proyecto activo”. El refactor a sub-toolbar de proyecto se completa en Fase 2.
  • Marcar proyectos con path inválido en rojo al arrancar, con opciones de re-elegir o eliminar.
  • Sin afectar PTYs todavía — el manager sigue siendo global. Solo seleccionás un proyecto visualmente y el breadcrumb refleja eso.

Fase 2 Terminales por proyecto

Objetivo: el set de terminales cambia con el proyecto activo.

  • Refactor: TerminalManager deja de ser singleton. Se crea uno por proyecto on-demand.
  • Nuevo TerminalRouter en app/ que monta/desmonta managers.
  • cwd de cada nueva terminal usa project.path por defecto.
  • Badge de “terminales vivas” en el sidebar.
  • “Run in all” limita su scope al proyecto activo.

Fase 3 Notas por proyecto

Objetivo: las notas siguen al proyecto.

  • notes-persistence recibe projectId como parámetro.
  • Migración de notas existentes a un proyecto “Default” o a un workspace nuevo.
  • Auto-save al cambiar de proyecto (flush antes del swap).

Fase 4 · opcional Quality of life

  • Command palette ⌘P para saltar a proyecto.
  • Drag & drop de carpetas del Finder al sidebar.
  • Drag & drop para reordenar / mover entre workspaces.
  • Color/ícono por workspace y por proyecto.
  • TerminalSpec.startupCmd: comando opcional auto-ejecutado al spawn (ej. pnpm dev).

Fase 5 · futuro Agentes

Cuando esto exista, el sidebar derecho (“agents flow”) se llenará con agentes vinculados al proyecto activo. Misma mecánica de swap. No diseñar todavía — definir bien Fase 1–3 primero.

10Decisiones tomadas

Resueltas. Estas son las reglas con las que arrancamos.

✓ Resuelta Estado actual: se descarta

Estamos en pruebas — las notas globales y configuración actual no se migran. El primer arranque tras el cambio parte de cero (cero workspaces, cero proyectos). El usuario tiene que crear el primer workspace + proyecto a mano.

Implicación técnica: al inicializar, si workspaces.json no existe o está vacío, se muestra un estado vacío con CTA grande: “Crea tu primer workspace”.

✓ Resuelta Proyectos simultáneos: sin límite

Todos los proyectos que hagan falta, todos vivos en paralelo (ya cubierto en § 02). No imponemos un máximo, no mostramos warnings de memoria, no “pausamos” proyectos viejos automáticamente. Si el usuario abre 30, la app respeta esa decisión.

Implicación técnica: no hay LRU, no hay quotas, no hay maxActiveProjects. El único límite real es la RAM de la máquina y eso es responsabilidad del usuario.

✓ Resuelta Orden: alfabético por defecto + drag & drop

Al crear un proyecto se inserta en su posición alfabética dentro del workspace. El usuario puede arrastrar para reordenar manualmente — esto fija un order explícito en el proyecto. Mismo comportamiento para workspaces a nivel raíz.

Implicación técnica: Project gana un campo order?: number opcional. Si no está definido → orden alfabético por name. Si está definido → respeta el número. Drag & drop reasigna order en todos los hermanos. Hay un menú “Ordenar alfabéticamente” en el ⋮ del workspace que limpia los order y vuelve al default.

✓ Resuelta Path: obligatorio y validado

Un proyecto siempre tiene un path. No se puede agregar un proyecto con path inválido — el formulario de creación lo rechaza antes de guardar.

  • Al crear: el file picker devuelve siempre una ruta real (filesystem picker nativo). Adicionalmente, antes de guardar se llama a un comando Rust fs_validate_path(path) que verifica: existe + es directorio + es readable. Si falla → toast de error y no se guarda.
  • Al editar “cambiar ruta”: misma validación.
  • Al arrancar la app, si un path persistido ya no existe (carpeta borrada externamente): el proyecto se muestra en rojo con ícono de advertencia y deshabilitado. Al clickearlo se ofrecen 2 acciones: Re-elegir ruta o Eliminar proyecto. No se silencia, no se borra solo.

✓ Resuelta Toolbar limpio · controles vinculados al proyecto

El toolbar superior actual tiene tres controles que no tienen sentido sin un proyecto activo: + Terminal, Run command in all…, Cols. Salen del toolbar global.

El nuevo toolbar global queda mínimo:

┌─────────────────────────────────────────────────────────────┐ │ Boom › Multiverse Admin [◧][▤][◨] │ └─────────────────────────────────────────────────────────────┘ ↑ breadcrumb (click = command palette) ↑ panel toggles

Los controles específicos del proyecto viven dentro del área del proyecto, sobre el grid de terminales:

┌─ Multiverse Admin ──────────────────────────────────────────┐ │ [+ Terminal] [Run in all…] Cols [3] │ ├─────────────────────────────────────────────────────────────┤ │ ┌──────────┐ ┌──────────┐ │ │ │ pnpm dev │ │ tests │ ← grid de terminales │ │ └──────────┘ └──────────┘ │ └─────────────────────────────────────────────────────────────┘

Si no hay proyecto activo (estado inicial o último proyecto eliminado), toda esta zona se reemplaza por un empty state:

┌─────────────────────────────────────────────────────────────┐ │ │ │ Sin proyecto activo │ │ Crea o selecciona un proyecto en el sidebar. │ │ │ │ [+ Crear workspace] │ │ │ └─────────────────────────────────────────────────────────────┘

Implicación técnica: el + Terminal, Run in all, y Cols dejan de ser hijos de .toolbar y pasan a ser hijos del template del project pane (sub-toolbar interno). La columna Cols se persiste por proyecto, no global — cada uno puede tener su propio layout.

Decisiones derivadas

  • Eliminar el último proyecto: no se auto-activa otro. La vista vuelve al empty state, aunque haya otros proyectos disponibles. El usuario elige conscientemente qué activar.
  • Workspace sin proyectos: se permite (recién creado). En el sidebar aparece con ＋ Añadir proyecto… directamente debajo del nombre, sin la fricción del hover.
  • Persistir el último proyecto activo al cerrar la app, y restaurarlo al abrir. Si ese proyecto tiene path inválido al restaurar → empty state (no auto-saltar a otro).
