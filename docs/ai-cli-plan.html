Plan de feature · polakapi

AI CLI en Terminales v0.1 · draft

Permitir que cada terminal de un proyecto arranque un CLI de AI (Claude Code, Codex, opencode) en lugar del shell por defecto. El objetivo: seleccionar el CLI de forma extremadamente sencilla, soportar mezclas (4× Claude, 2× Codex + 2× opencode) y que agregar un nuevo CLI en el futuro sea una línea en un registro.

1Contexto actual

Hoy cada terminal que se abre lanza el shell por defecto del sistema ($SHELL en macOS/Linux, %COMSPEC% en Windows) dentro de project.path. No existe ningún selector de shell ni CLI.

Toolbar actual

Cols

Lo importante: el plumbing ya existe. pty_spawn acepta command y args, y PaneCreateOptions ya los reenvía hasta Rust. El único bloqueo es que TerminalManager.addPane nunca los popula y el allowlist de Rust solo deja pasar shells POSIX.

Bloqueo en Rust is_allowed_shell en pty.rs rechaza todo lo que no sea sh/bash/zsh/fish/…. Lanzar claude como comando devuelve "shell not allowed: claude". Hay que extender el allowlist.

2Objetivos y escenarios

El feature debe resolver estos tres escenarios con la mínima fricción:

Escenario Acción deseada Métrica de fricción
4 terminales todas Claude Code Seleccionar Claude Code una vez, añadir 4 1 selección + 4 clics
4 terminales todas Codex Seleccionar Codex una vez, añadir 4 1 selección + 4 clics
2 Codex + 2 opencode Seleccionar Codex, añadir 2, seleccionar opencode, añadir 2 2 selecciones + 4 clics

Plus: al reabrir la app, los terminales persistidos deben respawn con su CLI correcto automáticamente.

3UX: Fila de chips + Badge por pane

Una fila de chips (segmented control) en la toolbar junto al botón + Terminal. El chip activo determina el CLI del próximo terminal. Un clic para cambiar, visible siempre. La selección es sticky: se mantiene hasta que el usuario cambia explícitamente de chip.

4Opción elegida

Fila de chips + Badge por pane Fila de chips visible siempre en la toolbar: un clic para cambiar de CLI, sin abrir menús. La selección es sticky —se mantiene hasta que se cambia—. Cada pane muestra un badge de color con su CLI. Los tres CLIs de AI + Shell son solo 4 chips, cabe holgado en la toolbar.

El badge por pane es display + respawn: muestra el CLI actual y, al hacer click, ofrece re-abrir el terminal con otro CLI. El respawn mata el PTY viejo y crea uno nuevo en el mismo cwd — es destructivo pero explícito (modal de confirmación si el terminal tiene output).

7Arquitectura

El núcleo es un registro único de perfiles CLI que es la fuente de verdad para todo: la fila de chips del toolbar, el badge del pane, la lógica de spawn, y la resolución al restaurar terminales persistidos.

Registro de CLI

// src/modules/terminal/cli-registry.ts

export interface CliProfile {
  id: string;          // estable, se persista en TerminalSpec.cliId
  label: string;        // texto del chip: "Claude Code"
  shortLabel: string;  // texto compacto del chip: "Claude"
  command: string;      // binario: "claude", "codex", "opencode"
  args?: string[];       // args fijos si aplica (ninguno por ahora)
  color: string;        // token CSS para el badge/dot: "orange" | "green" | "blue" | "slate"
  kind: "shell" | "ai-cli";
}

export const SHELL_PROFILE: CliProfile = {
  id: "shell",
  label: "Shell",
  shortLabel: "Shell",
  command: "",        // "" = resolver $SHELL en Rust
  color: "slate",
  kind: "shell",
};

export const AI_CLI_PROFILES: CliProfile[] = [
  { id: "claude",   label: "Claude Code", shortLabel: "Claude",   command: "claude",   color: "orange", kind: "ai-cli" },
  { id: "codex",    label: "Codex",       shortLabel: "Codex",    command: "codex",    color: "green",  kind: "ai-cli" },
  { id: "opencode", label: "opencode",    shortLabel: "opencode", command: "opencode", color: "blue",   kind: "ai-cli" },
];

export const ALL_PROFILES = [SHELL_PROFILE, ...AI_CLI_PROFILES];

export function resolveProfile(cliId?: string): CliProfile {
  return ALL_PROFILES.find(p => p.id === cliId) ?? SHELL_PROFILE;
}
Agregar un CLI nuevo = una entrada Para soportar un nuevo CLI en el futuro, se añade un objeto al array AI_CLI_PROFILES y su basename al allowlist de Rust. La fila de chips lo renderiza automáticamente. Nada más cambia.

8Persistencia

Se extiende TerminalSpec con un campo cliId que referencia un perfil del registro por id estable. No se persiste el command ni los args —se resuelven en runtime desde el registro—, lo que permite que el binario cambie de máquina o versión sin romper los terminales guardados. 

// src/modules/terminal/types.ts

export interface TerminalSpec {
  id: string;
  title?: string;
  cwd?: string;
  startupCmd?: string;
  cliId?: string;    // ← nuevo. undefined | "shell" = shell default
}

schemaVersion se mantiene en 1. cliId es opcional; los specs existentes sin cliId se interpretan como shell por defecto (resolveProfile(undefined) → SHELL_PROFILE). No requiere migración.

shallowEqualSpec en workspaces-reducer-terminals.ts:101 debe comparar cliId para que spec-changed dispare al cambiar el CLI de un pane. También TerminalManager.updateSpec en terminal-manager.ts:99-104 debe incluir cliId en su comparación identity.

9Cambios en Rust

is_allowed_shell en pty.rs:199 actualmente rechaza todo lo que no sea un shell POSIX. Se extiende con un allowlist separado para CLIs de AI. 

// src-tauri/src/pty.rs

/// Basenames permitidos para CLIs de AI. Distinto del allowlist de
/// shells porque estos no son shells interactivos —se lanzan dentro
/// de un PTY como un proceso normal.
const ALLOWED_AI_CLI_BASENAMES: &[&str] = &[
    "claude",
    "codex",
    "opencode",
];

fn is_allowed_command(cmd: &str) -> bool {
    is_allowed_shell(cmd) || basename_in(cmd, ALLOWED_AI_CLI_BASENAMES)
}

// resolve_shell (línea 184) → resolve_command, usa is_allowed_command
Decisión de diseño Los CLIs de AI se lanzan como el comando del PTY directamente (no dentro de un shell wrapper). command = "claude" arranca Claude Code como proceso del PTY en el cwd del proyecto. Los tres CLIs confirmados arrancan interactivos por defecto sin necesitar args.

Out of scope (futuro)

  • Detección de CLIs instalados: filtrar los chips a solo los CLIs presentes en $PATH (requiere un comando which en Rust).
  • Configuración de args por CLI: permitir que el usuario pase flags custom (ej. claude --model opus).
  • Plantillas de layout: "4× Claude" como preset guardable.
  • Autenticación por CLI: detectar si un CLI no está logueado y mostrar un hint.
  • Overflow menu: si se superan 6 CLIs, colapsar los menos usados en un chip "Más…" con dropdown.
