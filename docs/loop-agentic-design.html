/loop · proceso agéntico de 3 pasos

Diseño del flujo dentro de la ventana /loop. Borrador para iterar — todo es opinable.

PASO 1
Entender el objetivo
Chat con CLI para refinar el problema
→
PASO 2
Descomponer en fases
Lista de tareas · cada fase: .md + .html opcional
→
PASO 3
Ejecutar el loop
Secuencial o paralelo · 3–4 agentes por fase

CLIs disponibles: claudecodexopencode — cada agente puede usar uno distinto. Invocados como subprocesos shell desde Tauri.

Contexto de ejecución

El /loop siempre corre dentro del project activo del workspace. No existe loop "global".

Qué hereda del project

  • cwd = path del project (todos los CLIs se invocan desde ahí)
  • CLI por defecto sugerido (activeCliId) — el usuario lo puede pisar por agente
  • Configuración de git (branches, worktrees si aplican en el futuro)
  • Validación de path: si el project está marcado pathInvalid, /loop no arranca

Reglas de scope

  • Abrir /loop sin project activo: bloquea con prompt "elegí un project primero"
  • Todos los archivos del run viven en <project-path>/.loop/runs/<run-id>/
  • Cambiar de project con un run activo: el run sigue corriendo (atado al project original), pero la UI muestra "viendo project X · run activo en project Y"
  • Un mismo project puede tener múltiples runs históricos (carpeta runs/ con N entradas)

UI

  • Input grande de texto libre: "¿qué querés construir?"
  • Selector de CLI (claude / codex / opencode) — default heredado del project activo.
  • Chat panel: el LLM responde con preguntas clarificadoras; el usuario contesta inline.
  • Botón "consolidar" cuando el problema esté claro → produce problem.md.

Mecánica

  • Multi-turno con el CLI elegido. Prompt de sistema fijo (editable): "actuá como product engineer; hacé preguntas hasta entender alcance, restricciones, criterio de éxito".
  • Output final: problem.md firmado por el usuario (acepta/edita antes de pasar al Paso 2).
  • Persistido en el directorio del workspace activo: <project>/.loop/runs/<run-id>/01-problem.md.

Generación

  • CLI toma problem.md y produce una lista de fases (cada fase = una "tarea" del loop).
  • Por cada fase emite dos archivos:
    • NN-fase/logic.md — análisis técnico + pasos detallados
    • NN-fase/visual.html — mock visual (solo si aplica)
  • UI: árbol/lista de fases a la izquierda; tabs logic.md visual.html a la derecha.

Edición

  • Editor en línea para ambos archivos (Monaco o similar — el repo ya tiene ecosistema TS/Vite).
  • Botón "editar con AI" sobre cualquier sección: abre un mini-chat que toma el archivo + instrucción y devuelve diff.
  • Reordenar / agregar / borrar fases manualmente.
  • Preview del visual.html en iframe sandbox.

Agentes (compartidos entre modos)

Contrato: cada agente lee archivos del run y escribe solo su artefacto en disco. No hay conversación heredada entre CLIs distintos.

Agente Input Output Escribe FS? CLI / prompt
análisis logic.md + visual.html + knowledge previa analysis.md no configurable · prompts/analysis.md
implementación analysis.md + visual.html (ref) implementation.md + cambios reales en archivos sí configurable · prompts/implementation.md
revisor implementation.md + diff + criterio veredicto ok / retry+feedback (max 3 → ⚠ y sigue) no configurable · prompts/review.md
conocimiento todo lo anterior, incluye estado ⚠ si aplica knowledge.md (síntesis compacta) no configurable · prompts/knowledge.md
integrador (modo híbrido) todos los implementation.md + diffs del batch reporte de conflictos + knowledge.md consolidado del batch no configurable · prompts/integration.md

Modo A · Secuencial

Una fase a la vez. Cada fase enriquece a la siguiente vía knowledge.md.

┌─────────────────────────────────────────────┐ │ fase N │ │ │ │ logic.md + visual.html + knowledge[N-1] │ │ │ │ │ ▼ │ │ ┌────────────┐ │ │ │ análisis │ │ │ └─────┬──────┘ │ │ ▼ │ │ ┌────────────┐ │ │ │ implem. │◀──┐ │ │ └─────┬──────┘ │ retry + feedback │ │ ▼ │ │ │ ┌────────────┐ │ │ │ │ revisor │───┘ │ │ └─────┬──────┘ │ │ │ ok │ │ ▼ │ │ ┌────────────┐ │ │ │ conocim. │── knowledge[N] ─┐ │ │ └────────────┘ │ │ └─────────────────────────────────────┼───────┘ ▼ fase N+1 ...

✓ Coherente: cada fase ve lo que pasó antes.

⚠ Lento: tiempo total ≈ suma de fases.

⚠ Loop revisor → impl. puede ser infinito. Hace falta un máximo (¿3? ¿5?) + escape "aceptar igual / abortar".

Modo B · Híbrido (batched)

Cada fase declara dependsOn: [ids]. El sistema computa batches con sort topológico: las fases sin dependencias pendientes corren en paralelo, las dependientes esperan. Integrador entre cada batch.

batch 1 batch 2 batch 3 ┌── fase 1 ──┐ │ a → i → r │──┐ └────────────┘ │ ┌── fase 2 ──┐ │ ┌── fase 4 ──┐ │ a → i → r │──┼────▶│ a → i → r │──┐ └────────────┘ │ └────────────┘ │ ┌── fase 3 ──┐ │ ┌── fase 5 ──┐ │ ┌── fase 6 ──┐ │ a → i → r │──┘ │ a → i → r │──┴────▶│ a → i → r │ └────────────┘ └────────────┘ └────────────┘ │ │ │ ▼ ▼ ▼ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ integrador │ │ integrador │ │ integrador │ │ batch 1 │── K ─▶│ batch 2 │── K ─▶│ batch 3 │ └─────────────┘ └─────────────┘ └─────────────┘ │ ▼ merge final + reporte K = knowledge consolidado del batch (input del batch siguiente)

✓ Rápido sin perder coherencia: knowledge fluye entre batches.

✓ Conflictos detectados temprano: el integrador del batch N valida antes de arrancar N+1.

⚠ Requiere que el usuario (o el LLM) declare dependencias correctas en el Paso 2.

Decisiones tomadas

Decisión Resolución
Scope del loop cerrado Corre dentro del project activo (hereda cwd, CLI default, validación de path)
Modo paralelo cerrado Híbrido con dependsOn + batches topológicos. No worktrees, no patches. Sólo fases declaradas independientes corren en paralelo.
Declaración de dependencias cerrado Lista explícita dependsOn: [phaseId]. El LLM del Paso 2 propone, el usuario corrige en la UI.
Integrador en modo híbrido cerrado Uno por batch — valida no-conflictos y consolida el knowledge.md que entra al siguiente batch.
Tope del bucle revisor cerrado Max 3 intentos. Al límite, la fase pasa al agente de conocimiento con estado ⚠ sin aprobar — el knowledge anota la deuda.
Fallo de fase con dependientes cerrado La fase dependiente arranca igual, con el warning propagado en su contexto.
CLI por agente cerrado Matriz agente × CLI × modelo configurable por run. Ej: análisis = claude/opus-4.8, implementación = opencode/GLM-5.2, revisor = codex.
Contrato entre agentes cerrado Por archivos en disco, no por conversación heredada. Permite mezclar CLIs heterogéneos sin acoplar formatos.
Invocación de CLIs cerrado One-shot (claude -p, codex -p, opencode con flag equivalente). En el Paso 1 (multi-turno) cada turno serializa la conversación previa en el prompt.
Formato del knowledge.md cerrado Estructurado con plantilla fija (ver sección abajo). El agente de conocimiento e integrador llenan las mismas secciones.
Resume tras corte cerrado El run guarda state.json y se puede retomar buscando tareas incompletas. Granularidad: por agente. Outputs parciales se descartan.
Detección de parte visual cerrado Lo decide el LLM en el Paso 2. El usuario puede agregar/quitar manualmente en el editor.
Permisos de FS por agente cerrado Todos los agentes pueden escribir cualquier archivo. Mitigación: cada agente queda registrado con su diff propio, auditable en post-mortem.
Almacenamiento de perfiles cerrado JSON vía tauri-plugin-store (mismo patrón que workspaces.json). Archivo profiles.json en config dir de la app. No SQLite — overkill para 1–20 registros sin queries.
Alcance de perfiles cerrado Globales. No viajan con el repo. Son tus defaults personales.
Contenido de un perfil cerrado Sólo la matriz agente × CLI × modelo. max retries / budget / prompts son del run, no del perfil.
Default inicial sin perfil cargado cerrado Todos los slots arrancan en claude / opus-4.7. El usuario customiza desde ahí.
Built-in presets cerrado Ninguno. El usuario crea sus propios perfiles (botón "guardar como…" desde la matriz).
Comportamiento de overrides cerrado Temporales: los cambios sobre un perfil cargado aplican solo al run actual. Para persistir, "guardar" o "guardar como…" explícito.
Validación al cargar perfil cerrado Slot marcado en rojo si el CLI no está instalado o el modelo no existe. El usuario elige manualmente. No hay fallback automático.
Almacenamiento de prompts cerrado Globales editables en config dir de la app (prompts/*.md). Al crear un run, se copian al árbol del run. Edits del run no afectan globales (y viceversa para runs vivos).
UI para editar prompts cerrado Inline en el setup del Paso 3: sidebar con los 7 agentes + editor en el panel principal con dropdowns CLI/modelo arriba. Botones "↓ guardar como default global" y "↑ resetear a global" por agente.

Plantilla de knowledge.md

Estructura fija que rellenan tanto el agente de conocimiento (por fase) como el integrador (consolidado por batch). Secciones vacías permitidas — pero no removibles.

# Knowledge · <fase-id> o <batch-id> ## Decisiones tomadas - <decisión> — <razón breve> ## Archivos tocados - path/al/archivo.ts (+N / -M) agente=<impl|otro> - ... ## Supuestos - <cosa que se asumió que es cierta y la próxima fase puede confiar en> ## Gotchas - <cosa contra-intuitiva que la próxima fase debe saber para no repetir el error> ## Warnings (fases ⚠) - fase 04 quedó sin aprobar en 3/3 — el revisor pedía X; queda como deuda. - (vacío si todo aprobó) ## Próximos pasos sugeridos - <pista útil para el análisis de la fase/batch siguiente>

Modelo de resume

Si la app crashea o el usuario cierra el run, al reabrir /loop sobre ese project se detecta el run vivo y se ofrece retomar. Granularidad: por agente. Si el revisor estaba a medias, se descarta y se relanza desde cero (no se intenta retomar a media respuesta).

.loop/runs/<run-id>/state.json { "schemaVersion": 1, "runId": "a3f2", "mode": "hybrid", // o "sequential" "status": "running", // running | completed | aborted | crashed "currentBatch": 1, // null en modo secuencial "phases": { "01": { "status": "done", "lastAgent": "knowledge", "warning": false }, "02": { "status": "running", "lastAgent": "implementation", "retryAttempt": 2, "outputPartial": true }, "03": { "status": "pending" }, "04": { "status": "done", "lastAgent": "knowledge", "warning": true }, "05": { "status": "pending" } }, "integrators": { "batch1": "done" }, "lastHeartbeat": "<timestamp>" // si pasa > N min sin actualizar, asumir crash }

Vive en el config dir de la app (no se commitea con el repo). Mismo patrón de persistencia que workspaces.json (tauri-plugin-store con debounce + schemaVersion).

Validación práctica de que claude/codex/opencode se pueden invocar desde Tauri en modo one-shot. ✓ aprobado — sin riesgos técnicos bloqueantes.

Confirmado

  • Los 3 CLIs están instalados y soportan one-shot: claude -p, codex exec, opencode run
  • JSON output parseable en los 3 (claude: 1 doc; codex: JSONL + --output-last-message; opencode: stream JSON)

Forma de la integración

Pseudocódigo del comando Tauri que va a wrappear cada llamada de agente. No implementado todavía — es referencia para la fase de implementación.

// src-tauri/src/loop_cli.rs (nuevo, en la implementación) #[tauri::command] async fn run_loop_agent( cli: String, // "claude" | "codex" | "opencode" model: String, cwd: String, system_prompt_path: String, // path al prompt del agente (run-local) user_input: String, // contenido del input para esa invocación timeout_secs: u64, ) -> Result<AgentResult, String> { let args = match cli.as_str() { "claude" => vec!["-p", &user_input, "--model", &model, "--output-format", "json", "--append-system-prompt-file", &system_prompt_path], "codex" => vec!["exec", &user_input, "-m", &model, "--output-last-message", "/tmp/loop-out-<run>.txt"], "opencode" => vec!["run", &user_input, "-m", &model, "--format", "json", "--dir", &cwd], _ => return Err(format!("unknown cli: {}", cli)), }; let output = tokio::time::timeout( Duration::from_secs(timeout_secs), Command::new(&cli).args(args).current_dir(&cwd).output() ).await .map_err(|_| "timeout".to_string())? .map_err(|e| e.to_string())?; parse_output(&cli, &output) // normaliza al AgentResult común } // AgentResult { text, tokens_in, tokens_out, cost_usd, session_id, error }

Versiones validadas en el spike (24-jun-2026): claude 2.1.187, codex 0.142.0, opencode 1.17.9.
